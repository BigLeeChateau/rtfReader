## 一、问题陈述

**用户痛点:**

实际工作中经常需要打开页数非常多的 RTF 文件（1000+ 页、2000+ 页，甚至更多），但 Microsoft Word 打开这些文件非常卡顿，几乎无法正常使用。

**核心需求:**
1. 能够流畅打开超大 RTF 文件（2000+ 页）而不卡顿
2. 可以查看文件中已有的批注
3. 可以添加新的批注
4. 添加的批注能够被 Microsoft Word 识别（格式兼容）

**文件特征（由 /grill-with-docs 澄清）:**

- 文件类型：**临床试验 TFL**（Tables, Figures, Listings）
- 生成方式：SAS 等统计软件**自动生成**
- 内容构成：**除了图片就是表格**，文字占比极少
- 图片格式：SAS 默认输出 **EMF**（Enhanced Metafile），浏览器不能直接显示
- 批注规模：单文件可能有 **几百条批注**，来自多个审阅人
- 控制词特征：自动生成导致 RTF 控制词极度冗余

**问题根因分析:**

Word 打开大文件卡顿的原因在于它是一个全功能文字处理器，会同时执行：
- 全文档结构解析与排版
- 实时拼写检查与语法检查
- 修订历史与撤销栈维护
- 打印布局渲染
- 宏与插件加载

用户真正需要的是**"带批注功能的阅读器"**，而非"文字处理器"。

## 四、技术架构决策

### 3.1 实现方案对比

| 方案 | 描述 | 工作量 | 风险 | 推荐度 |
|------|------|--------|------|--------|
| **A. Tauri + Rust（选中）** | Rust 核心处理流式 RTF 解析与虚拟渲染，Web 前端做 UI。跨平台单二进制文件。 | M/L | 中 | ✅ 首选 |
| B. Electron + 现有 JS 库 | Electron 包装 JS RTF 解析器 + 虚拟滚动 | M | 中高 | ❌ JS 解析器内存模型难以保证大文件性能 |
| C. 原生按平台开发 | macOS 用 Swift，Windows 用 C# | XL | 高 | ❌ 双代码库维护 nightmare |

### 3.2 选型理由（Tauri + Rust）

- **性能是核心卖点**：Rust 的零成本抽象 + 内存安全，适合处理大文件解析
- **Tauri 二进制极小**：~5MB vs Electron ~150MB，用户安装无负担
- **虚拟滚动**：只渲染视口内内容，2000 页和 2 页一样流畅
- **跨平台**：一次开发，macOS + Windows 都能用

### 3.4 RTF 解析与渲染策略（关键决策）

**决策: 自定义方案 — 基于 rtf-parser lexer 改造，自行实现渲染**

背景调查结论:
- `rtf-parser` (crates.io 最活跃的 Rust RTF 库) **不支持流式解析**，会一次性将文件加载到内存
- 该库**不支持批注提取**
- 但 TFL 文件是**程序自动生成**的，结构高度规律，不需要处理通用 Word 文档的所有排版 edge case

**解析器策略:**
1. 借鉴 `rtf-parser` 的 lexer 控制词分词逻辑
2. 改造为 **yield 模式**：按块 emit（一个 Table/Figure/Listing 作为一个块）
3. 提取内容：文本、表格结构（行/列/合并单元格）、图注、批注控制词
4. 提取图片：识别 `\pict` 控制词，提取 EMF 二进制数据

**渲染策略（由 /grill-with-docs 澄清）:**
- **表格**: Rust 解析为结构化数据 → 前端用 **HTML/CSS Table** 渲染（利用浏览器排版引擎）
- **Listing**: 作为独立 **BlockType** 识别，实际仍按 HTML/CSS Table 渲染，未来可扩展特殊排版（横向滚动、更小字号）
- **图片**: EMF 数据 → Rust 后端转换为 **SVG** → 前端作为独立组件渲染；若 SVG 渲染异常或 `libemf2svg` 输出不完整，用 `resvg` 栅格化为 **300dpi PNG**；完全失败时显示占位框并提供导出 EMF
- **文本**: 直接渲染
- **虚拟滚动**: **块级粒度**（每个 Table/Figure/Listing 是一个块，而非行级）

**原型验证结论:**
- 自写流式解析器在 2000 页 RTF 上达到 **~10,000 页/秒**，峰值内存 **33 MB**，远超目标。
- HTML/CSS Table 渲染已通过 LibreOffice PDF 对比验证：行列、列宽、边框、对齐、中文字符均一致。
- EMF → SVG 转换在 2000 页 / 1000 张图文件上耗时 **0.027 秒**，全部成功。

**工作量重估:** 从最初估计的"改 300 行"上升为"写一个完整的 RTF→结构化数据转换器 + 批注管理器"

### 3.5 批注模型（关键决策）

**3.5.1 持久化: 嵌入式 RTF（Word 原生兼容）**

- 批注直接存储在 RTF 控制词结构中，使用 `\annotation`、`\atnauthor` 等控制词
- 优点：批注随文件移动，Word 可直接识别，无需 sidecar 文件
- 挑战：需要正确解析和修改 RTF 的组结构

RTF 批注结构示例:
```rtf
{\*\atnauthor Reviewer}
{\annotation
 {\*\atndate ...}
 Annotation text here.
}
```

规范要点:
- `\annotation` 和 `\atnauthor` 都是 destination 控制词
- **没有单个 group 同时包裹 author ID 和 annotation text** — 两部分是分离的
- 解析器需维护栈状态来处理这种结构

**3.5.2 批注权限与回复（由 /grill-with-docs 澄清）**

| 操作 | 自己的批注 | 别人的批注 |
|------|-----------|-----------|
| 查看 | ✅ | ✅ |
| 编辑内容 | ✅ | ❌ |
| 删除 | ✅ | ❌ |
| 回复 | — | ✅ |

- **回复模式**: 别人的批注原内容只读，但可以添加回复形成线程
- 回复数据与应用元数据嵌入 **RTF 自定义 destination** `{\*\rtfreader ...}`（Word 会忽略，但会随文件移动）
- 外部编辑检测：保存时记录 SHA-256 与 sidecar 备份，下次打开若 hash 变化则警告用户
- 单文件批注数量级: **几百条**

**3.5.3 作者身份识别（由 /grill-with-docs 澄清）**

- 首次启动 app 时要求配置用户名（用于批注作者标识），默认使用系统用户名
- 打开已有批注的文件时，从 `\atnauthor` 推断作者列表并填充为候选项
- 设置中可随时修改用户名
- 不同作者自动分配不同颜色：固定调色板 + 作者名哈希映射，保证同一作者跨文件颜色一致
- 自己的批注高亮显示；匿名批注显示为"未命名作者"（灰色）

## 七、数据流与交互设计

### 6.1 文档打开数据流

```
用户选择文件
    │
    ▼
Rust: 启动流式解析线程
    │
    ├──→ 解析文本块 ──→ 构建渲染区块索引（Table/Figure/Listing）
    │
    ├──→ 解析批注 ──→ 构建批注模型（全局字符偏移锚点）
    │
    └──→ 构建搜索索引（Tantivy，增量）
    │
    ▼
前端: 虚拟列表请求视口区块
    │
    ▼
Rust: 返回当前视口所需的渲染区块
    │
    ▼
前端: 渲染文本 + 高亮批注锚点
```

### 6.2 添加批注数据流

```
用户选中文本
    │
    ▼
前端: 记录选中文本的起止位置（字符偏移量）
    │
    ▼
用户输入批注内容
    │
    ▼
前端: 发送 {锚点位置, 批注内容, 作者, 时间} 到 Rust
    │
    ▼
Rust: 更新批注模型
    │
    ▼
前端: 实时渲染新批注标记
    │
    ▼
用户点击保存
    │
    ▼
Rust: 将批注序列化为 RTF 控制词
    │    首次保存 → 另存为新文件
    │    后续保存 → 覆盖已批注版本
    ▼
Word 验证: 用 Word 打开文件，批注可见
```

### 6.3 回复批注数据流（自定义扩展，Word 不可见）

```
用户点击别人的批注 → "回复"
    │
    ▼
前端: 输入回复内容
    │
    ▼
前端: 发送 {原批注ID, 回复内容, 回复者, 时间} 到 Rust
    │
    ▼
Rust: 存储到自定义数据结构（非 RTF 标准格式）
    │    保存为 sidecar JSON 或嵌入 RTF 自定义 destination
    ▼
前端: 在批注侧边栏显示为线程
```

### 6.4 块级虚拟渲染策略（由 /grill-with-docs 澄清）

TFL 文档天然分块：Listing 1 → Table 1 → Figure 1 → Table 2 → ...

```
文档总块数: N (50-200 个块)
每个块类型: Table | Figure | Listing | Text

策略:
1. Rust 解析时：识别块边界，记录块类型和预估高度
2. 前端虚拟滚动：只渲染当前视口 ± buffer 的块
3. 块内渲染：
   - 表格 → HTML/CSS Table（浏览器完整排版）
   - 图片 → 独立组件（EMF 转换后的 SVG/PNG）
   - 文本 → 直接渲染
4. 离开视口的块：DOM 节点回收或卸载，释放内存
```

## 八、技术风险与缓解

| 风险 | 状态 | 影响 | 缓解措施 |
|------|------|------|----------|
| RTF 格式复杂，edge case 多 | 🟡 中 | 解析器崩溃或渲染异常 | TFL 是程序生成，结构规律；优雅降级（跳过不认识控制词）；用实际文件测试 |
| 表格渲染偏差 | 🟢 已缓解 | 显示效果不如 Word | 原型已验证 HTML/CSS Table 可还原 r2rtf 表格；合并单元格待完善 |

| macOS 打包依赖 | 🟢 已缓解 | 目标机器无 Homebrew 无法运行 | `bundle-dylibs.sh` 已验证；注意 fontconfig 配置在极端环境可能缺失 |
