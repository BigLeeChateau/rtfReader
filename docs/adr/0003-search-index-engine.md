# ADR 0003: 全文搜索使用 Tantivy

## 状态

已接受

## 背景

应用需要支持 2000 页 TFL 的全文搜索，覆盖正文、表格内容、图注和批注文字。搜索需要在 200ms 内返回，且不能显著增加内存占用。

## 考虑的方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| **A. 自建最小倒排索引** | 轻量、无外部依赖 | 需要处理中文分词、前缀匹配、增量更新；长期维护成本高 |
| **B. SQLite FTS5** | 内置、支持中文、持久化 | 需要管理 SQLite 文件；增加部署复杂度 |
| **C. Tantivy** | 成熟 Rust 搜索引擎；支持中文分词；增量索引；性能好 | 增加依赖和编译时间；对简单场景略显重型 |
| **D. 前端 Fuse.js / MiniSearch** | 简单 | 需要把全部文本发往前端，违反大文件内存目标 |

## 决策

采用 **方案 C：Tantivy**。

- Rust 原生，与 Tauri 后端集成自然。
- 支持中文分词（可配置 jieba 或内置 tokenizer）。
- 增量索引：解析到一个块就更新索引，避免阻塞打开流程。
- 搜索结果带上下文预览和高亮。
- 索引文件持久化到磁盘缓存目录，重复打开可复用。

## 后果

- **正面**：功能上限高，未来可扩展模糊搜索、过滤、排序。
- **正面**：社区成熟，减少自建索引的边界情况处理。
- **负面**：编译时间增加，二进制体积增大。
- **负面**：需要学习和封装 Tantivy API，但工作量可控。
