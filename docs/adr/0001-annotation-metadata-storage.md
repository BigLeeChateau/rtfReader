# ADR 0001: 批注元数据与回复存储在 RTF 自定义 destination 中

## 状态

已接受

## 背景

本应用需要在 RTF 文件中存储两类非标准数据：

1. **批注回复线程**：Word 原生批注只支持单条文本，不支持回复嵌套。
2. **应用元数据**：例如作者颜色映射、最后保存时间、原始文件关联等。

这些数据必须跟随文件移动，否则用户复制或发送文件后会丢失回复。

## 考虑的方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| **A. Sidecar JSON**（与原文件同目录或固定目录） | 不污染 RTF；人类可读 | 文件移动后 sidecar 容易丢失；需要管理两个文件 |
| **B. 嵌入 RTF 自定义 destination** | 随文件移动；一个文件即可；Word 会忽略 | Word 保存时可能 strip 未知 destination；需要外部编辑检测 |
| **C. 写入标准批注文本** | 100% Word 兼容 | 无法表达回复线程结构；会污染批注可见内容 |

## 决策

采用 **方案 B**：将回复和元数据嵌入 RTF 自定义 destination `{\*\rtfreader ...}`，同时配套外部编辑检测机制。

- Word/LibreOffice 看到 `\*` 前缀的未知 destination 会安全忽略，不影响渲染。
- 保存时计算文件 SHA-256，存入 sidecar `checksums.json`；下次打开时对比 hash，变化则警告用户并允许从备份恢复。
- 文件名约定 `原文件名-annotated.rtf` 作为主检测，元数据作为辅助。

## 后果

- **正面**：文件可独立分享、移动、邮件发送，回复不会丢失。
- **正面**：不需要额外 sidecar 文件跟随用户文件系统。
- **负面**：如果用户用 Word 编辑并保存 `-annotated.rtf`，自定义 destination 可能被删除。我们通过 hash 检测 + 自动备份来降低损失。
- **风险**：Word 未来版本可能对未知 destination 行为变化，需要持续 roundtrip 测试。
